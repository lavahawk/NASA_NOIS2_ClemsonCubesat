# Background Worker Specification 2.3 Revision

## Purpose

This revision extends the background worker beyond point ingestion. The worker must continue ingesting VIIRS hotspot detections into `public.points`, and it must additionally derive and maintain active fire perimeter records from those detections.

Perimeters are generated by clustering eligible VIIRS points by location, converting each cluster into a polygon footprint, and merging that footprint into an existing active fire perimeter when the cluster is sufficiently close to an active fire.

## Scope

This revision adds:

- fire-perimeter generation from newly ingested VIIRS points
- active-fire merge behavior
- perimeter consolidation behavior when one cluster matches multiple active fires
- perimeter storage requirements in PostGIS
- point-to-perimeter linkage for provenance and rebuilds

This revision does not add:

- new FastAPI polygon endpoints
- perimeter archival beyond the active-fire definition in this document
- manual perimeter editing workflows
- perimeter-to-perimeter splitting logic

The baseline point ingestion contract in [background_worker_specification.md](/C:/Users/arche/OneDrive/Desktop/NASA_NOIS2_ClemsonCubesat/backend/background_worker/background_worker_specification.md) remains in force unless this revision explicitly overrides it.

## Worker Responsibilities

In addition to the existing point-ingestion responsibilities, the worker must:

1. Select perimeter-eligible points from each completed ingestion cycle.
2. Cluster those eligible points with DBSCAN using latitude/longitude positions.
3. Convert each cluster into a polygon footprint.
4. Compare each cluster footprint against active fire perimeters already stored in the database.
5. Merge the cluster into one or more active perimeters when the cluster is within the merge threshold.
6. Create a new fire perimeter when no active perimeter is close enough.
7. Update perimeter geometry, centroid, timestamps, and supporting linkage records in the database.

## Processing Order

For each polling cycle, the worker must execute the following sequence:

1. Complete the existing FIRMS availability check, query, normalization, and point insertion flow.
2. Identify the normalized points from that cycle that are eligible for perimeter generation.
3. Run DBSCAN on the eligible points from that cycle only.
4. Build one cluster footprint per cluster.
5. Load active fire perimeters from the database.
6. Order the generated clusters deterministically before applying them.
7. Apply clusters one at a time, not as one bulk merge operation.
8. Before applying each cluster, query the current active perimeter state, including any perimeter updates or consolidations created earlier in the same cycle.
9. For that cluster, find all active perimeters within the merge threshold.
10. If one or more qualifying active perimeters exist, merge them according to the consolidation rules in this document.
11. Otherwise create a new perimeter row for the cluster.
12. Record links between the perimeter and all points assigned to that cluster.

Point ingestion and perimeter generation are part of the same worker cycle, but perimeter generation occurs only after point normalization has succeeded.

The worker must not assume the original DBSCAN cluster set is final for perimeter purposes. Cluster application is iterative, and later clusters in the same cycle must be evaluated against the perimeter state produced by earlier cluster applications.

## Point Selection

Perimeter generation must not use every stored VIIRS point.

Eligible points are points from the current ingestion cycle whose normalized `confidence` value is:

- `n`
- `h`

The worker must exclude:

- `l`
- null or missing confidence values
- any unrecognized confidence values

Low-confidence points may still be stored in `public.points` under the existing ingestion specification. They are excluded only from the perimeter-generation pipeline.

Confidence comparison must be case-insensitive after trimming whitespace.

## DBSCAN Clustering

The worker must cluster the current cycle's eligible points with DBSCAN.

### Algorithm Contract

The clustering behavior must match the following implementation:

```python
import numpy as np
from sklearn.cluster import DBSCAN

EARTH_RADIUS_KM = 6371.0


def do_clustering_dbscan(data, max_thresh_km):
    """
    Cluster latitude/longitude points at one timestamp using DBSCAN.

    Parameters
    ----------
    data : sequence of (lat, lon)
        Point coordinates in degrees.
    max_thresh_km : float
        Maximum spatial separation in kilometers for points to belong
        to the same cluster.

    Returns
    -------
    list[int]
        Cluster id for each input point.
    """
    if not data:
        return []

    points = np.asarray(data, dtype=float)

    if len(points) < 3:
        return list(range(len(points)))

    points_rad = np.radians(points)

    labels = DBSCAN(
        eps=max_thresh_km / EARTH_RADIUS_KM,
        min_samples=1,
        metric="haversine",
        algorithm="ball_tree",
    ).fit_predict(points_rad)

    return labels.astype(int).tolist()
```

### Parameters

- `eps` must be derived from a configurable clustering threshold in kilometers.
- the default clustering threshold must be `5.0 km`
- `min_samples` must be `1`
- `metric` must be `haversine`
- `algorithm` must be `ball_tree`

### Rationale

`min_samples=1` is required so isolated points still become valid single-point clusters rather than DBSCAN noise.

`metric="haversine"` is required because the clustering input is latitude/longitude on the earth's surface rather than planar XY coordinates.

`algorithm="ball_tree"` is not semantically required, but it is the required implementation choice for this worker revision.

## Cluster Footprint Geometry

Each DBSCAN cluster must be converted into a polygon footprint before merge checks are performed.

### Canonical Geometry Method

For each point in the cluster, the worker must:

1. create a point geometry in SRID 4326
2. cast it to `geography`
3. buffer it by `375 meters`
4. cast the buffered result back to geometry

The cluster footprint must then be computed as the unary union of all buffered point geometries in that cluster.

The resulting perimeter geometry must be stored as `geometry(MultiPolygon, 4326)`. If PostGIS returns a `Polygon`, the worker must normalize it to `MultiPolygon` before storing it.

### Why a Fixed Buffer Is Required

This revision requires a polygon result even for a single-point cluster. A fixed 375-meter footprint approximates the nominal VIIRS pixel scale and avoids point or line geometries for small clusters.

## Active Fire Definition

A fire perimeter is considered active when:

- `updated_at >= now_utc - interval '5 days'`

Only active perimeters participate in merge checks.

Perimeters older than that threshold remain stored in the database but must not be considered merge targets for new clusters.

Perimeters with `merged = true` must also be excluded from merge targeting even if they still fall inside the active-fire time window.

## Perimeter Merging

### Merge Threshold

A cluster must be merged into an existing active fire perimeter when the minimum geodesic distance between:

- the cluster footprint geometry
- the active perimeter geometry

is less than or equal to `5.0 km`.

Distance must be evaluated with PostGIS geography-based measurement, not centroid distance and not planar degree distance.

### Iterative Cluster Application

DBSCAN output is only the initial grouping step for the current cycle. It is not the final authority on which detections belong to the same fire object.

Two or more clusters produced by DBSCAN may still belong to the same fire when they are each close enough to the same existing fire perimeter, or when one cluster updates or creates a perimeter that causes a later cluster in the same cycle to fall within the merge threshold.

For that reason, clusters must be applied iteratively. After each cluster is merged or inserted, the resulting perimeter state becomes the input state for the next cluster comparison in the same cycle.

The worker must use a deterministic cluster-processing order so repeated runs over the same input produce the same perimeter results.

### Candidate Selection

Every active perimeter whose geometry is within the merge threshold of the cluster footprint is a merge candidate.

If exactly one active perimeter is within the merge threshold, the worker must merge the cluster into that perimeter.

If more than one active perimeter is within the merge threshold, all candidates must be merged into a single new fire object.

### Merge Behavior

When exactly one existing perimeter is assigned to a cluster, the worker must:

1. union the existing perimeter geometry with the cluster footprint geometry
2. normalize the result to `MultiPolygon`
3. recompute the centroid from the updated geometry
4. update `updated_at` to the worker cycle timestamp
5. update `latest_detection_time` to the maximum acquisition time among the perimeter's existing points and the newly assigned cluster points
6. increase `detection_count` to reflect the total linked points
7. insert any missing point-to-perimeter linkage rows for the cluster points

### Perimeter Consolidation Behavior

When more than one active perimeter is within the merge threshold of a cluster footprint, the worker must create a new canonical perimeter row rather than choosing one existing row as the survivor.

In that case, the worker must:

1. create a new perimeter row
2. set `merged = true` on every previous perimeter row that participated in the consolidation
3. compute the new perimeter geometry as the union of:
   every candidate perimeter geometry and the current cluster footprint geometry
4. normalize the resulting geometry to `MultiPolygon`
5. recompute the centroid from the new geometry
6. set `created_at` on the new row to the cycle timestamp
7. set `updated_at` on the new row to the cycle timestamp
8. set `first_detection_time` on the new row to the earliest acquisition time across all points linked to the merged candidate perimeters and the current cluster points
9. set `latest_detection_time` on the new row to the latest acquisition time across all points linked to the merged candidate perimeters and the current cluster points
10. set `detection_count` on the new row to the total number of distinct linked points across all merged candidate perimeters and the current cluster points
11. create linkage rows from the new perimeter to every point linked to the merged candidate perimeters and every point in the current cluster

The previous perimeter rows must remain in the database for traceability, but once `merged = true` is set they are historical records only and must not be selected as future merge targets.

### New Perimeter Creation

When no active perimeter is within the merge threshold, the worker must create a new perimeter row with:

- `created_at` set to the cycle timestamp
- `updated_at` set to the cycle timestamp
- `first_detection_time` equal to the earliest acquisition time in the cluster
- `latest_detection_time` equal to the latest acquisition time in the cluster
- `detection_count` equal to the number of linked points in the cluster
- `geom` equal to the cluster footprint
- `centroid` derived from the stored geometry
- `merged` set to false by default

## Storage Contract

The existing `public.points` table remains unchanged.

This revision adds two new worker-owned tables:

- `public.fire_perimeters`
- `public.fire_perimeter_points`

### `public.fire_perimeters`

| Column | Type | Nullability | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | not null | Primary key. |
| `created_at` | `timestamptz` | not null | UTC creation time. |
| `updated_at` | `timestamptz` | not null | UTC last-merge time. |
| `first_detection_time` | `timestamptz` | not null | Earliest linked point acquisition time in UTC. |
| `latest_detection_time` | `timestamptz` | not null | Most recent linked point acquisition time in UTC. |
| `detection_count` | `integer` | not null | Count of linked points. |
| `merged` | `boolean` | not null | `false` for active canonical perimeters, `true` after the perimeter has been superseded by a later consolidation. |
| `geom` | `geometry(MultiPolygon, 4326)` | not null | Canonical perimeter geometry. |
| `centroid` | `geometry(Point, 4326)` | not null | Query helper derived from `geom`. |

### `public.fire_perimeter_points`

| Column | Type | Nullability | Notes |
| --- | --- | --- | --- |
| `fire_perimeter_id` | `uuid` | not null | Foreign key to `public.fire_perimeters(id)`. |
| `point_source_key` | `text` | not null | Foreign key to `public.points(source_key)`. |
| `linked_at` | `timestamptz` | not null | UTC linkage timestamp. |

The composite primary key must be:

- `fire_perimeter_id`
- `point_source_key`

The linkage table is required so perimeter geometry can be audited or rebuilt from the underlying points if needed.

### GeoJSON Storage Policy

The worker must not store a separate canonical GeoJSON polygon column.

Canonical storage is the PostGIS `geom` column. GeoJSON must be derived from `geom` at read time when needed by another service.

## Indexing Requirements

The migration set for this revision must create at minimum:

- a GiST index on `public.fire_perimeters.geom`
- a GiST index on `public.fire_perimeters.centroid`
- an index on `public.fire_perimeters.updated_at`
- an index on `public.fire_perimeter_points.point_source_key`

## Timestamp Policy

All perimeter-related timestamps must use UTC.

- `created_at`, `updated_at`, and `linked_at` are worker-managed UTC timestamps
- `first_detection_time` and `latest_detection_time` are derived from linked point `acquisition_time` values, which are already UTC

## Database Behavior

### Point Immutability

Rows in `public.points` remain immutable source observations. Perimeter generation must not modify point rows after insertion.

### Perimeter Mutability

Rows in `public.fire_perimeters` are mutable summary records and may be updated on later worker cycles as new clusters are merged into them.

If a perimeter row is superseded during a multi-candidate consolidation, that row must be retained with `merged = true` and must never be used again as the active canonical fire object.

### Idempotency

Repeated polling must remain safe.

- duplicate VIIRS points are still prevented by `public.points.source_key`
- duplicate perimeter-point links must be prevented by the composite primary key on `public.fire_perimeter_points`
- a point that is already linked to a perimeter must not produce a duplicate linkage row

This revision assumes a single worker instance is the only writer for the perimeter tables.

## Migration Contract

The worker migration system must be extended to create and maintain the new perimeter tables and indexes.

A new ordered SQL migration must:

1. create `public.fire_perimeters`
2. create `public.fire_perimeter_points`
3. create the `merged boolean not null default false` column on `public.fire_perimeters`
4. create the required indexes
5. add the required foreign-key constraints

Migration failure remains a startup failure. The worker must not continue into polling with only part of the perimeter schema applied.

## Error Handling

The worker must be tolerant of perimeter-generation failures at the cycle level.

- if perimeter clustering fails for a cycle, point ingestion for that cycle is still considered successful
- if perimeter geometry generation fails for one cluster, that cluster may be skipped, but the worker must continue processing other clusters in the same cycle
- if perimeter persistence fails after points have already been inserted, the worker must log the failure and retry implicitly on the next cycle

The worker must log enough structured information to identify:

- cycle timestamp
- number of perimeter-eligible points
- number of clusters produced
- number of new perimeters created
- number of existing perimeters updated
- number of clusters skipped due to error

## Operational Defaults

- perimeter clustering threshold: `5.0 km`
- perimeter merge threshold: `5.0 km`
- active-fire window: `5 days`
- point buffer radius for footprint generation: `375 meters`

## Out-of-Scope Follow-Up Work

The following items are intentionally not defined by this revision and should be specified separately if needed:

- API response shape for perimeters
- perimeter archival or explicit inactive status
- perimeter-to-perimeter consolidation when two existing active perimeters later overlap
- rebuilding all historical perimeters from the full points table
